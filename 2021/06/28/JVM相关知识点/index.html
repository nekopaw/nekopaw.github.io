JVM相关知识点

发表于 2021-06-28 | 分类于 知识点 ， Java ， JVM |

JVM

 HotSpot JVM把年轻代分为了三部分：1个Eden区和2个Survivor区（分别叫from和to）。默认比例为8：1，一般情况下，新创建的对象都会被分配到Eden区(一些大对象特殊处理),这些对象经过第一次Minor GC后，如果仍然存活，将会被移到Survivor区。对象在Survivor区中每熬过一次Minor GC，年龄就会增加1岁，当它的年龄增加到一定程度时，就会被移动到年老代中。

 在GC开始的时候，对象只会存在于Eden区和名为“From”的Survivor区，Survivor区“To”是空的。紧接着进行GC，Eden区中所有存活的对象都会被复制到“To”，而在“From”区中，仍存活的对象会根据他们的年龄值来决定去向。年龄达到一定值(年龄阈值，可以通过***-XX:MaxTenuringThreshold来设置)的对象会被移动到年老代中，没有达到阈值的对象会被复制到“To”区域。经过这次GC后，Eden区和From区已经被清空*。这个时候，“From”和“To”会交换他们的角色，也就是新的“To”就是上次GC前的“From”，新的“From”就是上次GC前的“To”。不管怎样，都会保证名为To的Survivor区域是空的。Minor GC会一直重复这样的过程，直到“To”区被填满，“To”区被填满之后，会将所有对象移动到年老代中。

有关年轻代的JVM参数

  1. -XX:NewSize和-XX:MaxNewSize
    用于设置年轻代的大小，建议设为整个堆大小的1/3或者1/4,两个值设为一样大。
  2. -XX:SurvivorRatio
    用于设置Eden和其中一个Survivor的比值，这个值也比较重要。
  3. -XX:+PrintTenuringDistribution
    这个参数用于显示每次Minor GC时Survivor区中各个年龄段的对象的大小。
  4. -XX:InitialTenuringThreshol和-XX:MaxTenuringThreshold
    用于设置晋升到老年代的对象年龄的最小值和最大值，每个对象在坚持过一次Minor GC之后，年龄就加1。

JVM垃圾收集器

  1. 新生代收集器：

    • Serial收集器：
       单线程执行,使用复制算法。在进行垃圾回收时，必须暂停其它所有工作线程。
       优点： 对于限定单个cpu的环境来说，可以获得最高的单线程手机效率
       缺点： 多线程效率底下
    • ParNew收集器
       使用方式： 
        通过 " -XX:+UseConcMarkSweepGC " 或者 " -XX:+UseParNewGC " 强制启动
  通过 " -XX:ParallelGCThreads " 来调整或者限制垃圾回收的线程数量
    • Parallel Scavenge收集器（吞吐量优先收集器）
       并行多线程收集容器，使用复制算法。目标是达到一个可控的吞吐量。
       吞吐量 = 程序运行时间 / （程序运行时间 + 垃圾收集时间 ）
       控制最大垃圾回收停顿时间：**” XX:MaxGCPauseMillis “**
        参数允许的值是一个“ >0 ”的毫秒数，收集器将尽可能的保证回收时间不超过设定值
  但并不是调小了就能让回收更快，GC停顿时间缩短是以牺牲吞吐量和新生代空间来换的
       设置吞吐量大小：**” -XX:GCTimeRatio “**
        参数的值应该是一个“ >0 && <100 ”的整数，也就是垃圾回收时间占总时间的比率
  如果设为19，则最大GC时间就占总时间的5%（即 1 / (1+19) ）
  默认值为99，允许最大1%（即 1 / (1+99) ）的垃圾收集时间
       开关参数：**” -XX:+UseAdaptiveSizePolicy “**
        开启该参数后就不需要手动指定新生代的大小（-Xmn）、Eden与Survivor区的比例（-XX:SurvivorRation）、晋升老年代对象年龄（-XX:PretenureSizeThreshold）；虚拟机会根据当前系统的运行情况收集性能监控信息，动态调整。这种调节方式称为GC自适应调节策略（GC Ergonomics)

  2. 老年代收集器：

    • Serial Old收集器
       单线程收集器，使用“标记-整理”算法
    • Parallel Old收集器
       是Parallel Scavenge收集器的老年代版本，使用“标记-整理”算法
      在注重吞吐量以及CPU资源敏感的场合，优先考虑Parallel Scavenge + Parallel Old

  3. CMS收集器

    • CMS(Concurrent Mark Sweep)收集器是以获取最短回收停顿时间为目标的收集器，使用（标记-清除）算法，整个过程分为4个步骤：  ①.初始标记（CMS initial mark）：标记一下GC Roots就能直接关联到对象，速度很快，期间会暂停其它线程。  ②.并发标记（CMS concurrent mark）：进行GC Roots Tracing的过程。  ③.重新标记（CMS remark）：修正并发标记期间因用户程序继续运作而导致标记产生变动的那一部分对象的标记记录，期间会暂停其它线程。  ④.并发清除（CMS concurrent sweep）
    • CMS收集器无法处理浮动垃圾（Floating Garbage：并发清除时用户线程产生的新垃圾），可能出现“Concurrent Mode Failure”而导致另一次Full GC的产生。
    • CMS收集器在JDK1.5中默认当老年代使用了68%空间后就会被激活，若应用中老年代增长不是太快可以适当调整参数：“ -XX:CMSInitiationgOccupancyFraction”以便降低内存回收次数，从而获得更好的性能。JDK1.6中默认值已提升至92%。
    • 若CMS运行时预留内存不够会出现“Concurrent Mode Failure”，这时虚拟机将启动后备预案：临时启用Serial Old收集器来重新进行老年代的垃圾回收，所以若是参数设置得太高容易导致大量的“Concurrent Mode Failure”，性能反而降低。
    • CMS有一个默认开启的参数：“-XX:UseCMSCompactAtFullCollection”（内存碎片合并整理），但是由于内存整理无法并发进行，会导致停顿时间变长。还有一个参数：“-XX:CMSFullGCsBeforeCompation”设置执行多少次Full GC后会执行一次内存碎片压缩（默认值为0）

  4. G1收集器

    G1是面向服务端应用的垃圾收集器。具有以下特点：

    • 并行与并发：G1能充分利用多CPU、多核环境下的硬件优势，使用多个CPU核心来缩短Stop-The-World停顿时间，不需要停顿java线程，而是并发的执行GC。
    • 分代收集：能独立管理整个GC堆。
    • 空间整合：G1从整体来看是基于“标记—整理”算法实现的收集器，从局部看（两个Region之间）是基于“复制”算法实现的。也就是说G1运作期间不会产生内存空间碎片。
    • 可预测的停顿：G1除了追求低停顿外，还能建立可预测的停顿时间模型，能让使用者明确指定在一个长度为M毫秒的时间片段内，消耗在垃圾收集上的时间不得超过N毫秒。

     在G1之前的其他收集器进行收集的范围都是整个新生代或者老年代，而G1不再是这样。使用G1收集器时，Java堆的内存布局就与其他收集器有很大差别，它将整个Java堆划分为多个大小相等的独立区域（Region），虽然还保留有新生代和老年代的概念，但新生代和老年代不再是物理隔离的了，它们都是一部分Region（不需要连续）的集合。


JVM调优

 老年代堆空间被占满，Full GC也无法回收触发异常：“java.lang.OutOfMemoryError:Java heap space”

 解决方案：

  1. 代码内的内存泄露可以通过一些分析工具进行分析，然后找出泄露点进行改善
  2. 优化代码和增加Survivor Space 等方法去优化

 持久代被占满，无法为新得class分配存储空间而引发异常：“java.lang.OutOfMemoryError: PermGen space”

 解决方案：

  1. 增加持久代的空间 -XX:MaxPermSize=100M
  2. 如果有自定义类加载的需要排除下自己的代码问题

 堆栈溢出，一般就是递归没有返回，或者循环调用造成异常：“java.lang.StackOverflowError”

 线程堆栈满，线程空间满了以后出现异常：“Fatal:Stack size too small”

 解决方案：

 增加线程栈大小，**-Xss2m**,但这个配置无法解决根本问题，还是要看代码部分是否有遗漏

 系统内存被占满，线程数量达到一定程度操作系统无法分配资源出现异常：“java.lang.OutOfMemoryError:unable to create new native thread”

 解决方案：

  1. 重新设计系统减少线程数量
  2. 线程数量不能减少的情况下，通过-Xss减小单个线程大小。
